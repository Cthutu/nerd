# Unit Test Framework

This folder contains sub-folders for unit tests.  The sub-folders have different
tests within them:

- `language` - tests for the language, such as parsing and type checking, IR
  generation, and code generation.
- `error` - test cases for error handling, such as syntax errors, type errors,
  and runtime errors.
- `commands` - command-level regressions for public compiler subcommands.

## Language Tests

In the `tests/language` folder, there are `.t` files that contain test cases for
the language.  Each test case is an example of code that should compile and run
successfully.

The `.t` files are divided into sections, each section separated by a `¬`
character.  The first section is also the actual code to test, and since the
compiler will stop at the `¬` character, it can be compiled like a normal source
file.

Below is a list of the sections in a `.t` file:

- The first section is the code to test.  This is the actual code that will be
  compiled and run.
- The second section is the expected return value of the program.  This is the
  value that the program should return when it is run.
- The third section is the expected output of the program.  This is the output
  that the program should produce when it is run.
- The fourth section is the IR that the compiler should generate for the code in
  human-readable form.
- The fifth section is the C code that the compiler should generate for the code
  in the first section.  This is what is passed to `clang` to compile the
  program.

## Error Tests

In the `tests/error` folder, there are `.e` files that contain test cases for
error handling.  Each test case is an example of code that should produce an
error when compiled.

The `.e` files are divided into sections, each section separated by a `¬`
character.  The first section is also the actual code to test, and since the
compiler will stop at the `¬` character, it can be compiled like a normal source
file.

The compiler can produce error messages in two formats: human-readable and JSON.
The unit tests in the `tests/error` folder use the JSON format, which is easier
to compare against expected output.

Below is a list of the sections in a `.e` file:

- The first section is the code to test.  This is the actual code that will be
  compiled and should produce an error.
- The second section is the expected error message in JSON format.  This is the
  error message that the compiler should produce when it encounters the error in
  the first section.

Unlike the language tests, there can be multiple tests in a single `.e` file,
and each test is separated by a `¬` character.  Each test consists of two
sections: the code to test and the expected error message in JSON format.

## Unit Test Runner

The unit tests can be run using the `test` command in the CLI.  This command
will run all the tests in the `tests` folder and report the results.  It will
iterate through the supported test folders and run each test case, comparing the
actual output to the expected output and reporting any discrepancies.

For the purposes of helping write the tests, if the IR generation or code
generation sections are left blank, the test runner will output the actual IR or
C code generated by the compiler, which can then be copied and pasted into the
test case.  This makes it easier to write tests without having to manually run
the compiler and capture the output.
